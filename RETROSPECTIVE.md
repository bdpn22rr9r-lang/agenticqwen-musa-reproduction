# 项目复盘

## 阶段：第一阶段（训练链路验证 + 首批数据交付）

时间：2026-07-09 ~ 2026-07-10

目标：在摩尔线程 MUSA GPU 上跑通 Qwen2.5-7B LoRA SFT 链路，并交付首批审校数据。

## 结论

**链路验证成功，数据初稿交付，但模型质量尚未证明。**

- ✅ 环境：容器 + MUSA + Torch-MUSA + Llama-Factory 全部就绪。
- ✅ 链路：单卡 LoRA 训练 → 权重保存 → 推理，全程无 NaN（在关闭 SDPA + 对齐 dtype 后）。
- ✅ 数据：80 训练 + 20 评测，类别配比与去重自验 PASS。
- ✅ 仓库：数据交付包上传 GitHub，可远程复现。
- ⚠️ 能力：3 step / 80 条样本只能证明「链路通」，不能证明「领域能力提升」；当前回答主要体现基础模型能力。

## 做得好的

- **数值问题定位到根因**：没有在 NaN 上反复盲目调参，而是通过「基础模型单独推理正常 → LoRA 异常 → 一步诊断定位 SDPA → 权重扫描确认」的对照实验，精准锁定 SDPA 反向传播，再用「训练 bf16 / 推理 fp16 不一致」解释残余 NaN。
- **数据策略审慎**：没有把 MechQA 自动标注数据直接拿训，而是发现并记录了它的系统性标注噪声，坚持人工核验、不合格即淘汰。
- **容器/镜像保护到位**：全程保留长期容器，危险 Docker 操作一律先报影响范围再确认，未误删任何环境。
- **官方资料优先**：每一步都对照摩尔线程官方文档，记录环境差异，不假装环境一致。
- **可复现**：把数据/配置/脚本沉淀成 GitHub 仓库 + 远程脚本，链路可一键复现。

## 需要改进的

- **样本量太少**：3 条 → 80 条仍是验证级，离领域能力证明差很远。第二阶段必须扩到 30–100 条经审校样本。
- **终端中文乱码**：PuTTY 远程中文显示为乱码，虽不影响模型语义，但影响人审阅。应固定 UTF-8 终端配置。
- **HF 网络受限**：远程无法直连 huggingface.co，数据下载依赖 GitHub 回退，全量 MechQA 暂未导入。
- **无固定评测**：当前评测集只有 20 条且偏定性，缺自动化指标（准确率/拒答率等）。
